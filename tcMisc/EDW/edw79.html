 CDATA attributes in P4

The following tables list all the attributes defined in TEI P4 as having a datatype of CDATA. For each attribute, it supplies at a minimum the name of the element or attribute class (attribute classes are distinguished by an initial asterisk; determining which elements are members of those classes is left as an exercise for the reader), and also any text currently supplied in P4 as value description for this attribute.

We have manually separated these 309 named attributes into three categories, `content textual', `textual', and `non-textual'.

The first category contains attributes which, because they may contain content, might include in their value some non-Unicode characters. It is this category of attributes that the Character Encoding Work Group believes must be expressible as elements, for there is way to encode a non-Unicode character within them as attributes. There are twenty-five of these attributes.

The second category contains attributes for which, even though it is not reasonable to expect that non-Unicode characters would be needed in their values, it is reasonable to expect for some other reason that it is advantageous to express their information as child elements rather than as attributes, usually in order to describe features about the information, e.g. language. The Character Encoding Work Group has recommended that these attributes be expressible as elements precisely for this reason: so that this information can have language, certainty, etc., associated with it. There are two of these attributes.

For these first two categories, in addition to the attribute name, the element or class, and the descriptive text from P4 (if any), the suggested disposition for this attribute in P5 is listed.

The third category contains attributes whose values are intended to be, or at least might plausibly be, restricted to symbolic values, i.e. which will only ever contain a single Unicode name token, a string of such tokens, numeric values, or boolean values. At a later revision of this document, this category may be further subdivided, in line with the recommendations of the TEI Meta work-group.

Appendix A Proposed Solutions for reg of the various Name elements

The following possibilities may well be applicable to several other content textual attributes, as well, but were initially dreamed up for handling reg of *names, *personPart, *placePart, *temporalExpr, <orgDivn>, <orgName>, <orgTitle>, and <orgType>.

  • Move reg to be <reg>, a child of whatever name element is being used, as a sibling of the PCDATA.
  • Give each of these naming elements two children: <reg> and a new element with the semantic ‘unregularized’, which could be named <literal>, <asIs>, <diplomatic>, <transcribed> or some such.
  • Move reg to be <regularized>, the second child of the new <choice> element, where the first child is the <name>, <persName>, <placeName>, whathaveyou.
  • Change reg to be a pointer type attribute (i.e., an XPointer). Provide, as a child of <encodingDesc>, a place to put a set of regularized names.

The first, although perhaps the simplest to conceive and easiest to execute, is quite distasteful for how it equates something at the PCDATA level with a child element. The second solves this problem, but leaves us with the problem that TEI-aware software will need to know about processing the children of each such naming element. The third solves this problem by using the proposed <choice> element as the `wrapper' for all such cases, continuing its semantic for processing — use only one of my children. The fourth removes the problem from the spot of encoding the name in the instance entirely; while it has the disadvantage of the confusion that is inevitable when a level of indirection is added, it has the major advantage of facilitating the use of a single `authority list' of names that is external to the current file.
